# LMForge v0.1 — Walkthrough & Developer Guide

Welcome to **LMForge**. This document describes everything implemented in the v0.1 core release, design decisions for developers, and operational guides for interacting with the orchestrator.

## What is LMForge?

LMForge is a hardware-aware, platform-agnostic local LLM orchestrator written in Rust.

Unlike single-engine apps (like Ollama), LMForge is an intelligent proxy. It probes the host machine's hardware, automatically selects the optimal backend inference engine (e.g. `oMLX` for Apple Silicon, `SGLang` for NVIDIA), installs the engine, downloads models in the optimal format (`mlx`, `gguf`, `safetensors`), and multiplexes API requests.

If you are developing an app (like a RAG pipeline or DocIntel), you point it to `http://localhost:11430/v1` and LMForge guarantees that it runs identically and optimally across any developer's hardware — no engine-specific code, no format-specific downloads.

## System Architecture

LMForge operates through an orchestrated pipeline:

1. **Hardware Probe** — Uses `sysinfo` to detect OS, CPU architecture, GPU vendor, and unified memory/VRAM. Result cached in `~/.lmforge/hardware.json`.
2. **Engine Registry (`engines.toml`)** — Driven by configuration. Cross-references hardware against known optimal engines, selecting by priority and compatibility.
3. **Engine Adapter Protocol** — Each engine (oMLX, SGLang, llama.cpp) implements a typed Rust trait. The adapter decides whether it handles the model pull natively or defers to LMForge's Rust downloader.
4. **Model Flow** — The CLI accepts generic aliases (e.g., `qwen3.5-4b`) or HuggingFace repos and translates them to exact endpoints in the format the active engine requires.
5. **Daemon Lifecycle** — `lmforge start` spawns the engine as a child process. It writes PID files, conducts health checks, exponential-backoff restarts, and proactive port-cleanup on startup.
6. **API Proxy** — An Axum server at `:11430` exposes OpenAI-compatible and Ollama-compatible endpoints, proxying to the engine at `:11431`.

## Key Features Shipped in v0.1

### Core CLI

| Command | Description |
|---------|-------------|
| `lmforge init` | Detect hardware, select engine, install if needed, scaffold `~/.lmforge/` |
| `lmforge pull <model>` | Download from HuggingFace. Resume-capable. Emits SSE progress. |
| `lmforge run <model>` | Interactive REPL with thinking-mode rendering |
| `lmforge start [--model <name>]` | Start daemon + engine |
| `lmforge stop` | Stop running daemon |
| `lmforge status` | Show engine and model status |
| `lmforge models list` | List installed models with sizes and capability flags |
| `lmforge models remove <name>` | Delete model from disk and index |
| `lmforge models unload` | Free VRAM immediately without removing files |
| `lmforge clean` | Disk audit: orphans, logs, HuggingFace cache duplicates |
| `lmforge logs` | View or tail daemon and engine logs |

### Interactive REPL (`lmforge run`)

Terminal chat interface with:
- `rustyline` for readline history and multiline input
- SSE stream parsing with live token rendering
- **Thinking Mode** — `<think>` blocks (Chain of Thought) rendered in dim-italic `\x1b[2;3m` while the model reasons, then the final answer in normal text

### Thinking Toggle — Ollama-Compatible

LMForge accepts the Ollama-standard `think: true/false` at the top level of any `/v1/chat/completions` request body. Internally it translates this to `chat_template_kwargs: {"enable_thinking": ...}` before forwarding to oMLX. The `think` field is stripped before forwarding so the engine never sees an unknown key.

```json
{ "model": "...", "messages": [...], "think": true }
```

The explicit `chat_template_kwargs` form is still accepted and takes precedence — direct callers that already use it are unaffected.

Any app written for the Ollama API works with LMForge with zero changes.

### Thinking — Non-Streaming `reasoning_content`

For non-streaming requests with `think: true`, LMForge internally forces a streaming request to the engine, accumulates `delta.reasoning_content` and `delta.content` chunks separately, then assembles a proper non-streaming response:

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "The answer is 4.",
      "reasoning_content": "The user asked 2+2. That equals 4."
    }
  }]
}
```

This matches the DeepSeek API contract and guarantees `reasoning_content` is always correctly separated — regardless of whether the engine's own non-streaming endpoint populates it.

### Streaming — `delta.reasoning_content`

For streaming requests with `think: true`, oMLX emits `delta.reasoning_content` and `delta.content` in separate SSE chunks natively. LMForge proxies them through unchanged:

```
data: {"choices":[{"delta":{"reasoning_content":"step 1..."}}]}
data: {"choices":[{"delta":{"content":"The answer is 4."}}]}
data: [DONE]
```

DocIntel and other clients just send `think: true` and read `delta.reasoning_content` — same interface as any Ollama-compatible app.

### Model Management and Disk Cleanup

`lmforge clean` provides a full disk audit before any destructive action:

- **Orphaned directories** — model dirs on disk not recorded in the index (typically interrupted downloads)
- **Stale index entries** — index entries whose directories no longer exist on disk
- **Log files** — accumulated engine and daemon logs
- **HuggingFace cache duplicates** — `~/.cache/huggingface/hub/` entries for models already downloaded into `~/.lmforge/models/`

```
lmforge clean              # interactive prompts per category
lmforge clean --dry-run    # audit only, no changes
lmforge clean --all --yes  # clean everything without prompts
lmforge clean --logs       # logs only
lmforge clean --partial    # orphaned dirs only
lmforge clean --hf-cache   # HF cache duplicates only
```

`lmforge models unload` stops the engine process and frees VRAM immediately. Model files remain on disk. The daemon stays alive at `:11430`. Reload any model with `lmforge models switch <name>`.

### Engine Adapter Protocol

Each engine implements a Rust trait with two methods:

- `pull_model` — Returns `Ok(true)` if the adapter handled the download (SGLang via `huggingface_hub.snapshot_download`), `Ok(false)` to defer to LMForge's Rust downloader (oMLX, llama.cpp), or `Err(e)` on failure.
- `start` — Spawn the engine process and return a handle.

oMLX and llama.cpp use LMForge's Rust downloader which writes directly to `~/.lmforge/models/`, bypassing `~/.cache/huggingface/`. SGLang uses native Python pull with `local_dir` pointed at the LMForge models directory, also bypassing the HF cache.

### Startup Cleanup (Proactive Orphan Kill)

On every `lmforge start`, before hardware probe or engine spawn, LMForge:

1. Reads `lmforge.pid` — sends SIGTERM, waits 3s, then SIGKILL to any stale daemon
2. Reads `engines/*.pid` — SIGKILLs any stale engine processes
3. For any port still occupied, uses `lsof` to identify and kill the holder
4. Waits until both `:11430` (API) and `:11431` (engine) are confirmed free before proceeding
5. Returns a clear error if either port cannot be freed after 10s

This replaces the previous reactive approach (detecting port conflict only after starting the engine) and eliminates the zombie-daemon failure mode entirely.

### Background Services (`lmforge service`)

For production or always-on setups:

- **macOS** — `com.lmforge.daemon.plist` via `launchd`
- **Linux** — `lmforge.service` via systemd user daemon

## Data Directory Layout

```
~/.lmforge/
├── models/
│   └── {model-id}/          ← one subdirectory per model
│       ├── config.json
│       ├── tokenizer_config.json
│       └── *.safetensors / *.gguf
├── engines/
│   └── {engine-id}/
│       └── venv/            ← Python venv for SGLang/oMLX
├── logs/
│   ├── lmforge.log
│   ├── engine-stdout.log
│   └── engine-stderr.log
├── models.json              ← model index (atomic writes via .tmp rename)
├── hardware.json            ← cached hardware profile
├── config.toml              ← user configuration
├── engines.toml             ← engine registry overrides
├── lmforge.pid              ← daemon PID
└── engines/{id}.pid         ← engine process PID
```
